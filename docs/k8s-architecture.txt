Script oral — Architecture Kubernetes                                                                                                                                      
                                                                                                                                                                             
  Introduction (30s)                                                                                                                                                         
                                                                                                                                                                             
  "Notre projet est déployé sur un cluster Kubernetes, en local avec Minikube et en production sur Google Cloud avec GKE Autopilot. On a 12 pods organisés en 3 couches."
                                                                                                                                                                             
  Couche 1 : Infrastructure — Helm Charts (1 min)           

  "Pour l'infra, on utilise 3 Helm Charts existants, comme recommandé dans le sujet :

  - PostgreSQL qui héberge 2 bases séparées : une pour Airflow (métadonnées des DAGs) et une pour MLflow (expériences, métriques, model registry)
  - MinIO, un stockage S3 local avec 2 buckets : un pour les artefacts MLflow (les modèles .pt) et un pour les données du dataset
  - Airflow avec 5 pods : le webserver pour l'interface, le scheduler qui exécute les DAGs, le triggerer, le DAG processor et StatsD pour les métriques internes"

  Couche 2 : Applicatif — Manifests K8s custom (1 min)

  "Pour la partie applicative, on a écrit nos propres manifests Kubernetes :

  - MLflow connecté à PostgreSQL et MinIO, qui sert de tracking server et model registry
  - L'API FastAPI qui embarque le modèle best.pt directement dans l'image Docker. Ça évite de dépendre de MLflow au démarrage en production. Elle expose 3 endpoints :
  /health, /predict et /metrics
  - La webapp Gradio qui communique avec l'API via le Service interne Kubernetes. L'utilisateur uploade une image et reçoit les détections de drones"

  Couche 3 : Monitoring (30s)

  "Pour le monitoring :

  - Prometheus scrape l'API toutes les 15 secondes et collecte 4 métriques : nombre de prédictions, erreurs, latence et détections
  - Grafana avec un dashboard pré-provisionné qui affiche le taux de requêtes, la latence p50/p95/p99 et le taux d'erreur"

  Exposition (30s)

  "Tous les services applicatifs sont exposés via des Services de type LoadBalancer. Sur Minikube on y accède via minikube service, sur GKE chaque service reçoit
  automatiquement une IP publique. On peut donc accéder à la webapp, l'API, MLflow, Prometheus et Grafana depuis n'importe quel navigateur."

  Conclusion (15s)

  "Au total, 12 pods, 3 Helm charts pour l'infra, 4 manifests custom pour l'applicatif, le tout déployable en 10 minutes avec les mêmes commandes en local ou sur le cloud."
